Authority Before Execution

Governance you can measure. Execution you can prove.

Autonomous systems don’t fail because they can’t reason. They fail because execution gets treated like a side effect instead of a controlled boundary.

Core invariant
No explicit authority at execution time → no state change
Authority must be present, scoped, and unexpired at the moment of commit.

What this is

A small control plane that separates proposal from execution. Agents can propose freely. Execution is gated by a time-bound authority artifact.

  • Fail-closed execution boundary
  • Typed decision contract
  • Deterministic allow/deny outputs
  • Decision-level evaluation summary

What this is not

This is not a platform. It intentionally avoids everything that would hide the invariant behind “features.”

  • RBAC / identity systems
  • Approval UI
  • Policy DSLs
  • Multi-agent orchestration
  • Production deployment concerns

Why this exists

Most systems

Analyze agent behavior after execution. Logs and postmortems reconstruct intent once damage is already done. Responsibility drifts, and governance becomes narrative.

This system

Treats execution as a controlled boundary. Authority is bound at runtime, and outcomes are evaluated as a system property. If authority isn’t present (or isn’t valid), the next state cannot exist.

Execution flow

User Task
  ↓
Agent (reasoning + proposal)
  ↓
Evaluation + trace (Opik)
  ↓
Authority gate
├─ Approved → Execute → Outcome traced
└─ Denied   → Block   → Denial traced
Execution boundary

Execution is the only place “state change” is permitted. Every commit attempt produces a structured, inspectable allow/deny result.

Authority properties
  • Explicit: never inferred
  • Scoped: covers only specific actions
  • Time-bound: expires automatically
  • Explainable: includes approver + reason

What gets measured

Allow rate
Allowed / total attempts
Shows how often execution is actually permitted under governance.
Denial breakdown
missing / expired / out-of-scope
Makes failure modes visible and comparable across runs.
Misuse signal
overbroad authority used
Flags governance anti-patterns where authority is too wide.
Why Opik matters here

Opik isn’t used as a dashboard accessory. It’s used at the boundary where truth exists: the execution commit and the evaluation summary.
