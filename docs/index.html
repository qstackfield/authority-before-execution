Authority Before Execution

An evaluation-first control plane for autonomous agents.

If explicit authority is not present at execution time, the next state does not exist.

Why This Exists

Most agent governance systems analyze behavior after execution. Logs are reviewed post-incident. Responsibility is inferred retroactively. Safety is narrated, not enforced.

This project takes the opposite stance: authority is bound before execution, responsibility is explicit, and outcomes are measured continuously.

The Core Invariant

No authority at execution time → no state change

Agents may reason freely. They may propose actions. But execution is a gated, observable, and measurable boundary.

Execution Flow

User Task
  ↓
Agent (Reasoning + Proposal)
  ↓
Evaluation + Trace (Opik)
  ↓
Authority Gate
├─ Approved → Execute → Outcome Traced
└─ Denied   → Block   → Denial Traced

Execution is impossible without a valid, scoped, time-bound authority artifact at the moment of commit.

What Is Measured

  • Execution allow / deny rate
  • Denial reason distribution
  • Expired or missing authority attempts
  • Out-of-scope authority usage
  • Overbroad authority detection

Governance quality is treated as a measurable system property, not a philosophical claim.

Experiments & Artifacts

This repository includes reproducible experiments and concrete artifacts:

  • Experiment Design - how governed vs ungoverned executions were compared
  • Sample Decision - the canonical execution input
  • Sample Execution Result - the resulting execution artifact

All executions are traced and evaluated using Opik, treating observability as a first-class primitive.

What This Is Not

This is not a full platform, UI, or orchestration layer. It intentionally avoids:

  • Role-based access control systems
  • Approval user interfaces
  • Policy DSLs
  • Multi-agent coordination
  • Production deployment concerns

The goal is clarity, not scale.

Hackathon Context

This project was built for the Encode “Commit To Change” AI Agents Hackathon with a focus on Best Use of Opik.

It demonstrates how evaluation and observability can be used to enforce governance before execution, not merely analyze failures afterward.
